## Schema 家族分类

10 个项目实际属于**三个 schema 家族**，按时间演化：

### 家族 A · 最新格式（3 项目，905 行）

包含项目: RIO_1, WTG, TXQ_1

**关键独有字段**:
- 主页链接 ⭐
- 粉丝数 ⭐
- 巡查状态、最近检查时间、已确认存活
- 数据回收情况
- 客户反馈、客户状态筛选

**字段命名**: 「曝光数」「阅读数」「互动数」（用"数"）

### 家族 B · 中间版本（5 项目，4,257 行）

包含项目: NRT_2, NRT_3, NUC_1, HXZ_QD, HXZ_FB

**关键独有字段**:
- 关键词、蓝词记录
- 项目阶段、打款金额、是否留存、理论金额结算
- 数据汇总、观众分析
- 父记录 2/3/4（飞书多维表格内部字段）
- 临时-评论修改对比

**关键缺失** ⚠️:
- 没有「主页链接」「粉丝数」

**字段命名**: 「曝光量」「阅读量」「互动量」（用"量"）

### 家族 C · 最老旧（2 项目，1,162 行）

包含项目: TGV_1, QSHG_1

**关键独有字段**:
- 大量 ad-hoc 日期化结算列（0907结算、0906评论结算）
- 备注（tier 标签藏在这里）⭐
- 产品贴数量、提问帖数量、私信/评论数量
- 二维码

**关键缺失** ⚠️:
- 没有「方向」字段
- 没有数据回收字段（曝光/阅读/互动全无）

## 关键发现

### 发现 1 · 「方向」字段是多维混编

10 个项目的「方向」字段，实际编码的维度不同：

| 项目 | 方向字段编码的维度 | 取值数 |
|---|---|---|
| RIO_1 | 内容形式 | 4 |
| WTG | 内容形式 + 渠道 | 5-6 |
| TXQ_1 | 内容形式 | 3 |
| NRT_2 | 内容方向 + 产品形式 + 身份 + 组合 | **15+** |
| NRT_3 | 内容方向 + 产品形式 + 身份 + 组合 | **18+** |
| NUC_1 | 用户场景 | 4 |
| HXZ_QD | 用户痛点 | 4 |
| HXZ_FB | 用户痛点 | 4 |
| TGV_1 | **N/A** | 0 |
| QSHG_1 | **N/A** | 0 |

**NRT 系列特别复杂**:
- 单标签例: "NRT 疗法引导"、"为爱助戒"、"戒烟贴"
- 组合标签例: "为爱助戒, 咀嚼胶"、"戒烟贴, 戒烟科普"
- 这种组合**单字段塞不下** —— 必须 schema 拆解

**这是 D-003 决策的数据依据**。

### 发现 2 · 「发布笔记」字段是 intent 金标

B 家族和 C 家族的「发布笔记」字段值（已归类到 intent）:

| 飞书原值 | 出现项目 | 数量 | 对应 intent |
|---|---|---|---|
| 流量帖 | NRT, NUC | ~1800 | traffic |
| 直给笔记 | NRT, NUC | ~1200 | conversion |
| 钓鱼帖 | TGV | ~200 | traffic |
| 科普贴 | TGV, NUC | ~150 | educational |
| 截图 | TGV | ~50 | other |
| 大字报 | TGV | ~30 | other |
| 小红书下单 | TGV | ~80 | conversion |
| 淘宝下单 | TGV | ~20 | conversion |

**直接用作 `intent` 字段映射，无歧义**。

### 发现 3 · 「状态」字段是 tier 金标（A/B 家族）

A 和 B 家族「状态」字段的取值分析：

| 飞书原值 | 出现频率 | 映射 tier |
|---|---|---|
| 无水花 | ~2700 | 趴 |
| 爆贴 | ~210 | 爆 |
| 大爆贴 | ~50 | 大爆 |
| 爆贴预备 | ~85 | 预备 |
| 风控中 | ~22 | 风控 |
| 评估中 | ~600 | 未知 |
| 未发布 | ~400 | (跳过) |
| 待发布 | ~280 | (跳过) |

**人工 tier 标注精度高** —— 和数据数值高度一致（爆贴样本的互动中位数 ~600+，无水花样本中位数 ~30）。

### 发现 4 · TGV_1 的「备注」字段藏着 47 个爆款金标 ⭐

TGV_1 没有数值数据，但「备注」字段含 tier 标签：

| 备注值 | 数量 | 映射 |
|---|---|---|
| 新爆 (含空格变体) | 47 | tier='爆' |
| 淘汰 | 305 | tier='趴' |
| 删0 (含 "他自己删了") | 64 | tier='删除' |
| 其他 / 空 | 359 | tier=null |

**这是 D-007 决策的数据依据** —— TGV_1 从 archive-only 升级到主表。

### 发现 5 · 「随贴评论」是被低估的资产

A 和 B 家族都有「随贴评论」+ B 家族有「随贴评论素人」字段，估算总样本量：

- A 家族评论样本: ~700
- B 家族评论样本（含素人评论）: ~2000
- **总评论样本: ~2,700**

这些评论包含：
- 控评植入的话术
- 真实素人评论（高质量）
- 评论 → 爆款的相关性数据

**单独成表（comments），不冗余存 notes 主表**。

### 发现 6 · B 家族粉丝数缺失（2300 条）

A 家族每条笔记都有「粉丝数」字段，B 家族**全部缺失**。

补录方案：
- 自动: 用 publish_url 调小红书爬虫 API
- 手动: 投放执行从原始账单查找

成本估算: 2300 条 × 5 秒每条 / API rate limit ≈ 4-8 小时（含校验）

### 发现 7 · 「观众分析」字段 B 家族大部分空

B 家族有「观众分析」字段，但**实际填写的极少**（<10%），且填写格式不一致（有的是 JSON、有的是描述性文本）。

结论：当前真实 audience 数据基本没有，**必须先用 LLM 推断 + 蒲公英真实数据补充**（D-008）。

## 跨项目的"低度对齐"线索

虽然 schema 不同，但有几个字段在大部分项目都有，且语义一致：

- **文案** → `raw_content` （所有项目）
- **发布时间** → `publish_time` （所有项目）
- **反馈链接** → `publish_url` （所有项目）
- **素人编号** → `note_id_suffix` （所有项目）

这意味着至少可以做：
- 跨项目去重（同一笔记不同投放）
- 时间维度分析（季度 / 月度趋势）
- URL 唯一性校验
