       │     │                    │                                                   │
       │     │              v_flywheel_lesson_cards(馆员读的视图)                     │
       │     │                                                                        │

**四层系统**(docs/01):L1 Core(标注+存储,**当前主力**)· L2 Predictor(发布前预测,`prepublish_evaluations`)·

## 5. 本会话定位的 4 个真根因(避免后人再踩)

### 5.1 馆员返回 `[]` —— 不是 cache_control,是【模型 env 名配错】
症状:autowriter 调馆员拿不到卡。曾误判为 cache_control 兼容(已加纯 system 重试 fallback,PR#46,但**不是**根因)。
**真根因**:Railway librarian 的 `FLYWHEEL_LIBRARIAN_MODEL` 没设成中转站通道实际支持的模型名 → LLM 调用静默失败 → 回空。
设对后缓存 0→1、馆员立刻正常。**教训**:三个服务三个不同的模型 env 名(§3.1),逐个核对。

### 5.2 backfill 反复 502 / "死很多次" —— 不是 worker 崩/OOM,是【Railway 边缘单请求 ~5min 超时】
症状:batch=40~50 时 ~4.5min 后 HTTP 502;曾误判为 worker OOM(后台一看 8vCPU/8GB 没崩、注解还在后台落库)。
**真根因**:Railway **边缘代理对单个 HTTP 请求 ~5min 硬超时**;大 batch 跑超 5min → 边缘掐断回 502,worker 其实没崩、
后台把这批标完了(标注幂等)。**修复**:batch 默认 50→**12**(每批 ~3min)、**硬上限 15**(>15 直接报错拦下,不靠自觉)。
→ **以后给任何表灌 essence,batch 留默认 12 就稳,别手贱调大。**

### 5.3 backfill 整步 `exit 5`、重试从没机会跑 —— bash `errexit` 撞 jq
worker 偶发重启吐空/坏响应时,`ok=$(jq …)` 解析失败在 `bash -e` 下触发 errexit 直接杀整步、绕过重试循环。
**修复**:loop 步首 `set +e`,靠显式检查 `$code/$ok` 决定 retry/exit;重试窗放宽到 6×60s。

### 5.4 worker 长任务被健康检查重启 —— 阻塞子进程占住事件循环
**修复**:`await run_in_threadpool(_run, ...)` 跑阻塞 subprocess,健康检查不再被饿死。

> 另:`v_flywheel_sync_status` 的 pending_* 曾误报(把永不被同步的行算"待同步")。已对齐各 sync 真实过滤
> (`tier_source <> '数值推断'`、publish_time 12 个月内、synthetic 规则、aw 侧复用权威候选视图),WTG pending 3→0。

## 7. 运行手册速查(怎么操作)

> **流程铁律(用户明确要求,务必遵守)**:① 不往**已合并**的 PR/分支推提交;② 先合 PR 再从 **main** 跑;
> ③ 不在未合并分支上擅自触发线上运行而不告知。新工作都开**基于最新 main 的新分支**。

**灌满一张表的 essence**(初次/补量):
GitHub → Actions → "Backfill essence" → 填 `project`(如 `WTG_phase1`),**batch 留默认 12**,Run。幂等,可反复跑。

**接一张新表**:
Actions → "Onboard table" → 填 project_id/app_token/table_id → 出 `mappings/<proj>.yaml` 草稿 PR → **人审**合并 → 灌数。

**手动跑一次 daily-sync**:Actions → "Daily TV sync" → Run(可填单项目 / dry_run / skip_essence)。

**查飞轮状态(SQL)**:
```sql
SELECT * FROM truth_vault.v_flywheel_sync_status;          -- 各项目推送状态
SELECT count(*) FROM truth_vault.flywheel_lesson_annotations; -- 书架厚度
SELECT count(*) FROM truth_vault.flywheel_librarian_cache;    -- 馆员是否被调用过
```

**自测馆员(D-038)**:见 docs/19 §自测 curl(带 `X-...-Key`,POST librarian `/brief`)。

   eval_wtg 只拿**结构**当金标准 oracle,不断言草稿判断值。
