## R-017 · AutoWriter 依赖上限 + lockfile

### 为什么

现状 `autowriter/requirements.txt`:

```text
streamlit>=1.30.0
anthropic>=0.40.0
google-genai>=0.5.0
supabase>=2.0.0
```

只用 `>=` 没上限. 任意一次 `pip install -r requirements.txt` 都可能拉到新的
major 版本:

- **streamlit 2.0** (未发布, 但官方在准备) 计划改 session_state 行为
- **anthropic 1.0** 切换 Messages API 默认参数
- **google-genai 1.0** 改 model 客户端入参约定
- **supabase 3.0** 可能动 PostgREST builder 的 .schema() 链式 API

TV 通道 2 sync 完全靠 supabase-py 的 schema-aware client (`autowriter/db.py`
里的 `ClientOptions(schema="autowriter")`) 和 PostgREST embedded join
(`batches!inner(project_id)`). supabase-py 一升小版本就可能让这些查询
silently 失败. AutoWriter 自己的 anthropic / google-genai 调用同理. 没有
lockfile 时, 每次 fresh deploy 拉的依赖版本不一样, 出问题查不到根因.

### 怎么做

#### Step 1 · 加上限

`autowriter/requirements.txt` 改成:

```text
# 上限是经验值, 锁到下一个 major 版本前. 真的要升级 major 之前应该:
#   1. 看上游 changelog
#   2. 在 staging 跑 smoke test
#   3. 一次升一个, 不要堆一起
streamlit>=1.30.0,<2.0
anthropic>=0.40.0,<1.0
google-genai>=0.5.0,<1.0
supabase>=2.0.0,<3.0
```

#### Step 2 · 生成 lockfile

用 pip-tools (主流, 已经在 truth-vault scripts/ 用):

```bash
cd autowriter/
python -m pip install pip-tools
pip-compile requirements.txt -o requirements.lock --resolver=backtracking --strip-extras
```

或用 uv (更快, 推荐迁过去):

```bash
cd autowriter/
uv pip compile requirements.txt -o requirements.lock
```

`requirements.lock` 长这样 (pinned 完整版本号):

```text
# This file is autogenerated by pip-compile with Python 3.11
streamlit==1.40.2
anthropic==0.49.0
google-genai==0.8.0
supabase==2.18.1
# ...所有 transitive deps 也都 pin 死
```

#### Step 3 · 部署脚本读 lockfile

`autowriter/Procfile` / Dockerfile / 你部署用的脚本里, install 命令从:

```bash
pip install -r requirements.txt
```

改成:

```bash
pip install -r requirements.lock
```

#### Step 4 · CI 烟雾测试

`.github/workflows/ci.yml` 加一段:

```yaml
- name: install + smoke import
  run: |
    pip install -r requirements.lock
    python - <<'PY'
    # 不能跑 streamlit (需要 ScriptRunContext); 单独 import 验证依赖解析 OK
    import streamlit
    from anthropic import Anthropic
    from google import genai
    from supabase import create_client, ClientOptions
    # 关键: PostgREST schema-aware client 必须能 instantiate
    client = create_client(
        "https://placeholder.supabase.co",
        "sb_secret_placeholder",
        options=ClientOptions(schema="autowriter"),
    )
    assert hasattr(client, "schema"), "supabase-py changed schema() API"
    print("smoke import OK")
    PY
```

CI 失败时再去看新版有什么 breaking change.

### 升级流程 (以后想升 major 版本时)

```bash
# 不要直接改 requirements.txt 的上限, 单独跑实验:
cd autowriter/
python -m venv .upgrade-venv && source .upgrade-venv/bin/activate
pip install "supabase>=3.0,<4.0"
pip freeze | grep supabase  # 看具体新版号
python - <<'PY'
# 抄 step 4 的 smoke test 跑一遍
PY
# 跑过通过 -> 改 requirements.txt 上限 -> pip-compile -> commit
```

## R-018 · daemon thread → 持久 job worker

两个仓的设计 99% 一样. 表 schema 见 `autowriter-migrations/008_jobs_table.sql`
和 `sanshengliubu-patches/004_jobs_table.sql`. 下面写 Python 进程层 + UI
改造 + 部署.

### 状态机

```
   ┌──────────┐  worker.claim_one_job()
   │ pending  │ ─────────────────────────► claimed (transient, 不实际驻留)
   └────┬─────┘                                   │
        │                                         │ handler 开始执行
        ▲                                         ▼
        │ sweeper 把死 job 退回                  running (heartbeat 中)
        │ (heartbeat > timeout, attempts < max) ──┤
        │                                         │
        │                          handler 返回成功 │ handler 抛错
        │                                ▼         ▼
   ┌────┴─────┐                       success    failed
   │  failed  │ ◄────── attempts >= max_attempts 时 sweeper 转过来
   └──────────┘
        ▲
        │ user 从 UI 触发
   ┌────┴─────┐
   │cancelled │
   └──────────┘
```

`claimed` 这个状态实际上是 `claim_one_job()` RPC 内部从 pending 改到
running 的 transient 中间态; 用 SELECT ... FOR UPDATE SKIP LOCKED 一个事务
完成, 不会驻留. 列在状态机里只是为了万一 RPC 中途断网 / DB crash, 能用
sweeper 把孤儿 claimed 行回收.

### Worker 进程设计

新文件: `autowriter/worker.py` (sanshengliubu 那边: `sanshengliubu/worker.py`,
near-identical, 只是 schema 名和 kind 集合不同).

#### 1. 主循环

```python
"""
autowriter/worker.py

独立进程, 从 autowriter.jobs 表领取并处理 job. 不依赖 Streamlit 进程.

部署: 见 docs/10-sister-repo-followups.md § "部署". 至少跑 2 副本
(一个 OOM/crash 不影响另一个).

环境变量:
    SUPABASE_URL                   必填
    SUPABASE_SERVICE_ROLE_KEY      必填. service_role 绕 RLS, 必须设环境变量
                                   不要硬编码进镜像. 泄露立即 rotate.
    WORKER_ID_PREFIX               默认 "aw-worker"; 多机部署时建议带主机名
                                   `hostname-`, 方便排错时看 claimed_by 定位.
    WORKER_KINDS                   默认 "" (所有 kind). 用逗号分隔, 比如
                                   "generate_batch,quick_gen" 让这个 worker
                                   只领快任务, 别的 worker 跑 memory_refresh.
    POLL_INTERVAL_SECONDS          默认 2; 队列空时轮询间隔.
    HEARTBEAT_INTERVAL_SECONDS     默认 15; 处理中心跳频率.
    HEARTBEAT_TIMEOUT_SECONDS      默认 90; 超过这个时长没心跳 sweeper 回收.
"""
import os
import sys
import time
import socket
import signal
import logging
import threading
import traceback
from datetime import datetime, timezone, timedelta
from supabase import create_client, ClientOptions

logger = logging.getLogger("aw_worker")
logging.basicConfig(level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s · %(message)s")

# ── 配置 ─────────────────────────────────────────────────────────
WORKER_ID = f"{os.environ.get('WORKER_ID_PREFIX', 'aw-worker')}-{socket.gethostname()}-{os.getpid()}"
WORKER_KINDS = [k.strip() for k in os.environ.get("WORKER_KINDS", "").split(",") if k.strip()] or None
POLL_INTERVAL = int(os.environ.get("POLL_INTERVAL_SECONDS", "2"))
HEARTBEAT_INTERVAL = int(os.environ.get("HEARTBEAT_INTERVAL_SECONDS", "15"))
HEARTBEAT_TIMEOUT = int(os.environ.get("HEARTBEAT_TIMEOUT_SECONDS", "90"))

# Supabase 必须用 service_role. anon JWT 走 RLS 看不到别人的 job.
sb = create_client(
    os.environ["SUPABASE_URL"],
    os.environ["SUPABASE_SERVICE_ROLE_KEY"],
    options=ClientOptions(schema="autowriter"),
)

# 优雅退出: SIGTERM 时停止领新 job, 把当前 job 跑完
_shutdown = threading.Event()
def _on_signal(sig, frame):
    logger.info("received signal %s; draining and exiting", sig)
    _shutdown.set()
signal.signal(signal.SIGTERM, _on_signal)
signal.signal(signal.SIGINT, _on_signal)


# ── handler 注册表 ──────────────────────────────────────────────
HANDLERS = {}

def register(kind):
    """把 def my_handler(job, sb): ...  注册成 kind=X 的处理器."""
    def deco(fn):
        HANDLERS[kind] = fn
        return fn
    return deco


@register("noop")
def _noop_handler(job, sb):
    """Phase 1 部署验证用. 啥也不做, 立即 success."""
    return {"ok": True, "echo": job["payload"]}


@register("generate_batch")
def _generate_batch_handler(job, sb):
    """包现 autowriter/app.py:_queue_worker 的批量生成逻辑.

    payload schema:
        {"project_id": "...", "tactic": "...", "ai_engines": [...],
         "params": {...}}

    实现时把现 _queue_worker 的逻辑移过来, 把 st.session_state 写
    改成 _report_progress(job, pct, msg) (我们封装好的 helper).
    """
    from autowriter import generator    # lazy import 让 worker 进程不 import streamlit
    payload = job["payload"]
    # ... 调 generator.generate_batch(...) ...
    # 中间 update progress: _report_progress(job, 30, "已生成 3/10 个版本")
    # 返回 success result: 让 UI 知道 batch_id / item_ids 等
    return {"batch_id": "...", "item_ids": [...]}


@register("quick_gen")
def _quick_gen_handler(job, sb):
    """包现 _quick_gen_worker 的单版本快速生成."""
    # 类似上面


# ── 心跳 + progress helper ──────────────────────────────────────
def _heartbeat_loop(job_id, stop_event):
    while not stop_event.is_set():
        try:
            sb.table("jobs").update({"heartbeat_at": _now_iso()}).eq("id", job_id).execute()
        except Exception:
            logger.exception("heartbeat update failed for %s", job_id)
        stop_event.wait(HEARTBEAT_INTERVAL)


def _report_progress(job_id, pct, message=None):
    """Handler 调这个更新进度. UI 轮询同一个 job 行就能看到."""
    patch = {"progress_pct": pct}
    if message is not None:
        patch["progress_message"] = message
    try:
        sb.table("jobs").update(patch).eq("id", job_id).execute()
    except Exception:
        logger.exception("progress update failed for %s", job_id)


def _now_iso():
    return datetime.now(timezone.utc).isoformat(timespec="seconds")


# ── 处理一个 job ────────────────────────────────────────────────
def process_one(job):
    handler = HANDLERS.get(job["kind"])
    if handler is None:
        logger.error("no handler for kind=%s, marking failed", job["kind"])
        sb.table("jobs").update({
            "status": "failed",
            "finished_at": _now_iso(),
            "error_text": f"no handler registered for kind={job['kind']}",
        }).eq("id", job["id"]).execute()
        return

    # 起心跳 thread (仅心跳, 不跑业务; 这样长 LLM 调用不会让心跳跟着 block)
    stop_hb = threading.Event()
    hb_thread = threading.Thread(target=_heartbeat_loop, args=(job["id"], stop_hb), daemon=True)
    hb_thread.start()

    try:
        result = handler(job, sb)
        sb.table("jobs").update({
            "status": "success",
            "finished_at": _now_iso(),
            "progress_pct": 100,
            "result": result or {},
        }).eq("id", job["id"]).execute()
        logger.info("job %s (kind=%s) success", job["id"], job["kind"])
    except Exception:
        tb = traceback.format_exc()
        # attempts 字段 RPC 时已经 +1; 这里只看是否还能重试
        if job["attempts"] >= job["max_attempts"]:
            new_status = "failed"
        else:
            new_status = "pending"
            # 指数回退: 30s, 60s, 120s ...
            backoff = 30 * (2 ** (job["attempts"] - 1))
            patch_extra = {
                "next_retry_at": (datetime.now(timezone.utc) + timedelta(seconds=backoff)).isoformat(timespec="seconds"),
            }
        patch = {
            "status": new_status,
            "finished_at": _now_iso() if new_status == "failed" else None,
            "error_text": tb,
        }
        if new_status == "pending":
            patch.update(patch_extra)
        sb.table("jobs").update(patch).eq("id", job["id"]).execute()
        logger.exception("job %s (kind=%s) → %s", job["id"], job["kind"], new_status)
    finally:
        stop_hb.set()


# ── 死 job sweeper (每次主循环 idle 时机会顺手扫一次) ──────────
_last_sweep = 0.0
SWEEP_INTERVAL_SECONDS = 60

def maybe_sweep():
    global _last_sweep
    if time.time() - _last_sweep < SWEEP_INTERVAL_SECONDS:
        return
    _last_sweep = time.time()
    cutoff = (datetime.now(timezone.utc) - timedelta(seconds=HEARTBEAT_TIMEOUT)).isoformat(timespec="seconds")
    # 1. heartbeat 超时 → pending 重试
    res = sb.table("jobs").update({
        "status": "pending",
        "claimed_by": None,
        "claimed_at": None,
        "heartbeat_at": None,
        "started_at": None,
        "next_retry_at": _now_iso(),
        "error_text": f"heartbeat stale (cutoff={cutoff}); worker likely died, retrying",
    }).in_("status", ["claimed", "running"]).lt("heartbeat_at", cutoff).execute()
    n_recovered = len(res.data or [])
    if n_recovered:
        logger.warning("sweeper recovered %d dead jobs", n_recovered)

    # 2. attempts >= max_attempts 的 pending 行直接 failed (避免无限重试)
    # 这步用 RPC 比较稳, 这里给 SQL 形式; 实际用一个简短的 RPC 函数包.
    # (省略实现 — Streamlit 启动时跑一次 SQL 创建即可.)


# ── 主循环 ──────────────────────────────────────────────────────
def main():
    logger.info("worker %s starting; kinds=%s", WORKER_ID, WORKER_KINDS or "ALL")
    while not _shutdown.is_set():
        try:
            res = sb.rpc("claim_one_job", {
                "_worker_id": WORKER_ID,
                "_kinds": WORKER_KINDS,
            }).execute()
            jobs = res.data or []
            if not jobs:
                maybe_sweep()
                _shutdown.wait(POLL_INTERVAL)
                continue
            for job in jobs:    # RPC 返回 SETOF, 单次实际只有 0 或 1 行
                process_one(job)
        except Exception:
            logger.exception("worker main loop exception; sleeping before retry")
            _shutdown.wait(POLL_INTERVAL * 5)

    logger.info("worker %s exited cleanly", WORKER_ID)


if __name__ == "__main__":
    main()
```

```python
def show_job_status(job_id):
    """轮询一个 job 行, 画 progress bar. 调用方负责定时 st.rerun()."""
    res = (
        sb.schema("autowriter")
        .table("jobs")
        .select("status, progress_pct, progress_message, result, error_text, finished_at")
        .eq("id", job_id)
        .single()
        .execute()
    )
    j = res.data
    if j["status"] in ("pending", "claimed"):
        st.info("⏳ 任务排队中...")
    elif j["status"] == "running":
        st.progress(j["progress_pct"] / 100)
        if j["progress_message"]:
            st.text(j["progress_message"])
    elif j["status"] == "success":
        st.success("✅ 完成")
        if j["result"]:
            st.json(j["result"])
    elif j["status"] == "failed":
        st.error("❌ 失败")
        with st.expander("错误详情"):
            st.code(j["error_text"])
    elif j["status"] == "timeout":
        st.warning("⏱ 超时, 已自动重试")
    elif j["status"] == "cancelled":
        st.info("用户取消")
    return j["status"] in ("success", "failed", "cancelled")    # True = 终态


# 在页面里:
if "running_job_id" in st.session_state:

    if not is_done:
        time.sleep(2)
        st.rerun()
```

#### 3. 部署 (线上服务器, 不跑 docker)

##### 选项 A · systemd unit (推荐, 最稳)

适合阿里云 / 腾讯云 / AWS Lightsail / 任何标准 Linux VPS.

`/etc/systemd/system/autowriter-worker.service`:

```ini
[Unit]
Description=AutoWriter background worker
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=autowriter
Group=autowriter
WorkingDirectory=/opt/autowriter
EnvironmentFile=/etc/autowriter/.env    # SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY 等
ExecStart=/opt/autowriter/venv/bin/python -m autowriter.worker
Restart=always
RestartSec=5
# OOM 后自动重启; journal 日志限速防刷盘
StandardOutput=journal
StandardError=journal
SyslogIdentifier=autowriter-worker

[Install]
WantedBy=multi-user.target
```

部署:

```bash
sudo cp autowriter-worker.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now autowriter-worker
# 看日志: sudo journalctl -u autowriter-worker -f
# 重启:   sudo systemctl restart autowriter-worker
```

跑多副本 (推荐 ≥ 2 防单点):

```bash
sudo cp /etc/systemd/system/autowriter-worker.service /etc/systemd/system/autowriter-worker@.service
# 然后 enable: autowriter-worker@1, autowriter-worker@2
sudo systemctl enable --now autowriter-worker@1 autowriter-worker@2
```

单 batch 经常超 10 分钟, 不要走这条路.

### 迁移分 phase 走

**不要一刀切, 让飞轮在过渡期还能跑.**

**Phase 1** (1-2 天): 跑 008 migration → 部署 worker 进程, 但只注册 'noop'
handler. Streamlit UI 不动. 在 Supabase Dashboard 手工
`INSERT INTO jobs (kind, payload) VALUES ('noop', '{}')` 几次, 验证
worker 能领能跑能写回 success. 验证 OOM 后自动重启. 验证心跳 + sweeper.

**Phase 2** (3-7 天): 加 'generate_batch' handler. UI 加 feature flag
`USE_JOB_QUEUE` (env, 默认 false). 旧 thread 路径继续, 新 job 路径只在
staging / flag=true 时走. 用真实 batch 跑通.

**Phase 3** (生产开关): 把 flag 在生产打开. 监控一周: jobs 表里有没有
不应该的 failed / timeout / 长时间 pending. 监控 worker journal 日志
有没有 unexpected exception.

**Phase 4** (清理): 删除 app.py 里的 _queue_worker / _quick_gen_worker 函数和
对应 thread 启动代码. 删掉 feature flag. 自此 app.py 瘦 ~500 行.

### 监控指标 (上线后看的)

```sql
-- 队列健康
SELECT status, COUNT(*) FROM autowriter.jobs GROUP BY status;
-- pending 持续 > 100 表示 worker 太少, 加副本

-- 平均处理时长
SELECT kind, AVG(EXTRACT(EPOCH FROM (finished_at - started_at))) AS avg_sec
FROM autowriter.jobs WHERE status='success' AND finished_at > NOW() - INTERVAL '1 day'
GROUP BY kind;

-- 卡 running 的 (健康时应该都很短; 长期 >max_runtime_seconds 是 bug)
SELECT id, kind, started_at, NOW() - started_at AS elapsed, claimed_by
FROM autowriter.jobs WHERE status='running' AND started_at < NOW() - INTERVAL '10 min';

-- 失败率
SELECT kind,
    SUM(CASE WHEN status='failed' THEN 1 ELSE 0 END)::FLOAT / COUNT(*) AS fail_rate
FROM autowriter.jobs WHERE created_at > NOW() - INTERVAL '7 days'
GROUP BY kind;
```

#### B2. 手工把现有用户加进默认 workspace

migration 不能自动做这一步 (auth.users 在每个 project 状态不同). 跑完
migration **立即跑**:

```sql
-- 把所有现有 auth.users 都设为默认 workspace 的 owner. 后续可以细分 role.
INSERT INTO public.workspace_users (workspace_id, user_id, role)
```

#### B3. sanshengliubu 仓代码改造

##### B3.1 db/supabase_client.py

加 `current_workspace_id()` helper:

```python
def current_workspace_id(sb) -> str:
    """当前登录用户的主 workspace_id.

    多 workspace 用户走 owner 优先排序, 同 role 取 invited_at 最早的.
    单 workspace 用户直接返回那个 id.
    """
    user = sb.auth.get_user()
    if user is None or user.user is None:
        raise RuntimeError("not authenticated — cannot determine workspace")
    user_id = user.user.id
    res = (
        sb.table("workspace_users")
        .select("workspace_id, role, invited_at")
        .eq("user_id", user_id)
        .execute()
    )
    if not res.data:
        raise RuntimeError(
            f"user {user_id} is not a member of any workspace. "
            "Add them via INSERT INTO workspace_users(...) using service_role."
        )
    res.data.sort(key=lambda r: (
        {"owner": 0, "admin": 1, "member": 2, "viewer": 3}[r["role"]],
        r["invited_at"],
    ))
    return res.data[0]["workspace_id"]
```

##### B3.2 所有 INSERT 路径带 workspace_id

找出所有 `.table("projects").insert(...)`,
`.table("pipeline_runs").insert(...)`, 等等. 改造模式:

```python
# 旧:
sb.table("projects").insert({"name": "..."}).execute()

# 新:
sb.table("projects").insert({
    "name": "...",
    "workspace_id": current_workspace_id(sb),
}).execute()
```

如果项目用了 helper 类 (e.g., `db/projects.py:create_project()`), 把
workspace_id 注入放 helper 内部, 不要让所有 caller 自己加.

##### B3.3 TV 通道 1 sync 不需要改

理由: R-018 worker 落地之后, app.py 里 ~500 行 thread/queue 管理逻辑会被

- ❌ 不要先做大 PR. 一个文件一次, 一周一拆, 让 review 跟得上.
