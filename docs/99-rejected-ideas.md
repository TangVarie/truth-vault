# 99 · Rejected Ideas · 走过的弯路

## 为什么存在

每个项目都有一些**当时讨论过但被否决**的方案。这些方案如果不记录下来，6 个月后会有人（包括你自己 / 新窗口的 Claude）重新提出来 —— 然后再花一遍时间走相同的路。

这个文档是项目的"考古防火墙"。

> 一条铁律：发现自己/Claude 在重新提一个看起来"明显应该做"的方案前，**先来这个文档搜一下**。如果已经被拒过，看完原因再决定是否重提。

---

## R-001 · 用 RAG（embedding 检索）作为主要查询方式

**提出时间**: 2026-05 早期讨论

**方案**: 把所有历史笔记 embedding 化，存 pgvector。用户给一条新文案，cosine similarity 找 top-K 相似笔记，作为参考。

**听起来为什么诱人**:
- 工程简单（pgvector 配好就行）
- 业界主流（"AI + 营销" 工具几乎都是这套）
- 数据多就能 scale

**为什么被否决**:
- 用 RIO 一期 170 条数据实验：爆款（互动 500+）和趴帖（互动 3）的文案字面高度相似 —— embedding similarity > 0.85
- 长尾分布污染：170 条里 160 条趴，top-20 检索 95% 是趴
- 真正决定爆 vs 趴的因素**不在文案字面里**（账号当时流量包、评论区引爆、时段、KOL 等级…）
- Ziao 原话："RAG 本质是匹配，你怎么能确保匹配到的是精髓呢？"
- 即使能找到相似的爆款，新内容**模仿表层会被平台风控**，且容易丧失新意

**对应决策**: [D-002](../DECISIONS.md#d-002)

**什么时候可以重新考虑**:
- 阶段 3 (5k+ 数据后)，作为**辅助召回**（不是主查询），且只在"essence + audience 相似 + 爆款"的过滤集合里做 embedding 匹配
- 永远不会作为主查询方式

## R-002 · Schema 只做 surface 层，先跑起来再说

**提出时间**: 2026-05 早期讨论

**方案**: 先做最小 schema（标题、正文、互动数、tier），快速跑通，等数据多了再加 essence / audience 层。

**听起来为什么诱人**:
- 工程上简单
- 短期看起来"敏捷"
- MVP 思路

**为什么被否决**:
- Surface-only 模型 6-12 个月就因为时间漂移失效
- 历史数据后期回标 essence 的质量一致性会出问题（标注人员换、标准漂移）
- 取上得中 —— 起点决定上限
- Ziao 原话："这个东西在一开始就需要有个口子，不是说后来怎么怎么着"

**对应决策**: [D-001](../DECISIONS.md#d-001)

**什么时候可以重新考虑**: 不会。这是项目的根基决策。

## R-003 · 让 LLM 自由描述 essence（不闭集）

**提出时间**: 2026-05 中期讨论

**方案**: Essence 标注不限闭集，让 LLM 自由描述每条笔记的情绪杠杆和人性原型。

**听起来为什么诱人**:
- 表达力强，不会被词表"压死"信息
- LLM 能力越来越强，可以信任它

**为什么被否决**:
- 3000 条数据用自由描述跑出来会有 ~2800 种不同表述
- 跨样本不可比，无法做统计
- 没办法 GROUP BY 看"焦虑撬动"的爆款率
- 没办法做"两个项目的 essence 相似度"
- 跨产品策略迁移完全做不出来

**对应决策**: 受控词表方案（[05-controlled-vocab.md](05-controlled-vocab.md)）

**什么时候可以重新考虑**:
- 自由文本可以**附加**在闭集字段旁（如 audience.psychographic 的自由文本部分）
- 但主字段必须闭集 —— 这一条不会改

## R-004 · "方向" 字段直接当作内容形式分类

**提出时间**: 2026-05 早期讨论

**方案**: 飞书表的「方向」字段是天然的分类标签，直接当作 `content_format` 字段入库。

**听起来为什么诱人**:
- 不用做额外解析
- 项目经理已经填好了

**为什么被否决**:
- 「方向」字段实际编码 **3-4 个不同维度**:
  - RIO_1 的方向是内容形式（喝酒感受）
  - NUC_1 的方向是用户场景（术后恢复）
  - HXZ 的方向是用户痛点（持妆问题）
  - NRT_2/3 的方向是身份 + 产品形式 + 内容方向**三维混编**
- 直接当 content_format 用 → 跨项目无法对比
- NRT 还有 "为爱助戒, 咀嚼胶" 这种组合标签，单字段塞不下

**对应决策**: [D-003](../DECISIONS.md#d-003) —— Schema 层面拆解为多维（content_format / target_audience / user_pain_point / product_focus）

**什么时候可以重新考虑**: 不会。但 onboarding 时**保留原始「方向」字段**进 raw_extra，永远不丢。

## R-005 · 假设 A 家族（RIO/WTG/TXQ）是最老的格式

**提出时间**: 2026-05-18 数据审计中期

**方案**: 看到 RIO_1 字段比 NUC_1 少了「关键词」「父记录」等，下意识判断 RIO 是更老的、不完整的格式。

**听起来为什么诱人**:
- 字段少 = 不完整 = 老 ← 这个直觉错了
- 我也是这么想的，没多想

**为什么被否决**:
- Ziao review 时纠正：A 家族独有的「主页链接」「粉丝数」「巡查状态」「最近检查时间」「已确认存活」「数据回收情况」**全部是现代化特征**
- B 家族缺粉丝数 → 是早期飞书表演化阶段，运营管理体系还没建立
- 真正的演化顺序：C（最老）→ B（中间，加了方向 + 数据回收）→ A（最新，加了运营管理体系）

**对应决策**: [D-006](../DECISIONS.md#d-006)

**Implications**:
- 标准 schema 按 A 家族字段设计
- B 家族数据需要补录粉丝数（2,300 条）
- 不要"对齐到 B 家族" —— 之前的设计错了

**教训**: 跨项目的 schema 演化方向，必须问当事人确认，不要根据字段数推断。

## R-006 · TGV_1 只进 archive 表（因为缺数据）

**提出时间**: 2026-05-18 数据审计早期

**方案**: TGV_1 没有曝光/阅读/互动数据，没法判断 tier，归为"无 label" 数据，只进 archive 表作为文案语料库。

**听起来为什么诱人**:
- 没数值数据看起来就是"半成品"
- 进 archive 简化处理

**为什么被否决**:
- Ziao 提醒后重新审视 TGV_1 的「备注」字段，发现 **47 条人工"新爆"标注**
- 人工"爆"标注是**金标准**（运营标爆很谨慎）
- 47 爆 + 305 趴是干净的二分类训练数据
- 即使没有数值数据，二分类训练依然完全有效

**对应决策**: [D-007](../DECISIONS.md#d-007) —— TGV_1 进 notes 主表（数值字段允许 null）

**Implications**:
- C 家族特殊 mapping：tier 抽取从「备注」字段
- 训练数据池从 3,000 → 3,400（+13%）
- 增加 47 个爆款样本

**教训**: "缺数据"不等于"没价值"。再看一遍数据，看看人工标注藏在哪里。

## R-007 · 管家直接给出 P(爆) 概率或推荐改写方向

**提出时间**: 2026-05 中期讨论

**方案**: 既然有数据飞轮，让"智能管家"直接给出"这条 P(爆)=0.32"或者"建议改成 XX 方向"。

**听起来为什么诱人**:
- 用户体验好（直接给答案）
- LLM 看起来"更智能"

**为什么被否决**:
- 管家做内容判断 = 引入 LLM 幻觉
- 数据库的边界被破坏 —— "查数据"和"做判断"应该角色分离
- 判断权应该在 persona / critic / 人类写手手里
- "管家"角色就是查询和统计，不做价值判断

**对应决策**: [D-004](../DECISIONS.md#d-004)

**什么时候可以重新考虑**:
- 阶段 2 启动后，**判别式模型**给出 P(爆) —— 这是模型而不是 LLM 幻觉
- 但即使有模型，给出的也是"置信度+特征对比"，不是"建议改写为 XX"
- 改写建议永远由生成层（autowriter）和评判层（persona/critic）做

## R-008 · 只标新数据，老数据先放着

**提出时间**: 2026-05 中期讨论

**方案**: Essence 标注成本高，3400 条历史数据先不标，从新数据开始标，等三个月后看效果再决定。

**听起来为什么诱人**:
- 省钱
- "敏捷"思路
- 万一标注规则后面要改，老数据白标了

**为什么被否决**:
- 历史 3400 条**是爆款样本最集中的资产**（约 280-330 个爆款）
- 不标 = 模型阶段 2 没有充足正样本训练
- 越晚标，标注一致性问题越严重（标注的人/标准会变）
- 标注成本可控（¥1000-1500），不值得省

**对应决策**: [D-005](../DECISIONS.md#d-005) —— 历史数据必须回标 essence

**教训**: 一次性投入解决永久性资产问题，比分批投入解决重复性问题划算。

## R-009 · Audience 数据只用 LLM 推断

**提出时间**: 2026-05-18 audience 层讨论中

**方案**: 既然 LLM 能推断 audience，就不用接蒲公英真实数据了 —— 简化工程。

**听起来为什么诱人**:
- 不用做蒲公英数据导入
- 不用考虑客户合规
- 工程简单

**为什么被否决**:
- LLM audience 推断**没有校准基线** —— 错了也不知道
- 真实 audience 数据是帆谷的**独特优势**（其他 AI 营销工具拿不到）
- 校准闭环（LLM vs 真实数据 → 优化 prompt）是数据飞轮在 audience 层的具体实现
- Ziao 现在就能拉，不拉浪费

**对应决策**: [D-008](../DECISIONS.md#d-008)

## R-010 · 用 oransim 的因果建模思路重做

**提出时间**: 2026-05 早期（项目最初的触发）

**方案**: 看到 OranAi 的 oransim 项目，考虑借鉴它的因果数字孪生方法 —— 模拟反事实"如果当时改了 X 文案会怎样"。

**听起来为什么诱人**:
- 学术上先进
- "Counterfactual reasoning" 听起来很厉害

**为什么被否决**:
- oransim 是基于公开互联网数据训练的人格模型 —— 帆谷需要的是**真实投放数据闭环**
- 因果推断需要**至少 1k+ 实验数据**和 prompt 元数据，当前阶段还不到
- "持续提升"的真正杠杆是**回流数据**，不是模拟器
- 算法不是护城河，数据飞轮才是
- oransim 适合阶段 4，不是阶段 1 的入口

**对应决策**: 项目方向从"基于 oransim 的 persona 评估"转向"基于真实数据的 Truth Vault"

**什么时候可以重新考虑**: 阶段 4 (20k+ 数据)，借鉴 oransim 的方法做 CATE 估计。

## R-011 · 拒绝集成 sanshengliubu / autowriter，做独立产品

**提出时间**: 2026-05 中期讨论（隐含）

**方案**: Truth Vault 完全独立，自己做 UI 给用户用，不集成现有工具。

**听起来为什么诱人**:
- 边界清晰
- 不被现有工具拖累
- 可以单独商业化

**为什么被否决**:
- 帆谷的内容生产已经在 sanshengliubu + autowriter 跑通
- Truth Vault 的价值是**给现有工具注入决策 anchor**，不是替代它们
- 独立产品 = 重复造轮子（用户管理、UI、auth 都要重做）
- 工程资源稀缺，应该集中

**对应决策**: Truth Vault 定位为**数据基础设施**，向三个调用方提供 API（sanshengliubu / autowriter / 写手网络）

**Implications**:
- 不做面向客户的 UI（除了内部维护 Web UI）
- 工程栈和 sanshengliubu 对齐（Supabase + FastAPI）
- 阶段 1 验收指标包括"被 sanshengliubu 和 autowriter 调用"

## 一些"看起来对但还没拍板"的方案（议程区）

下面这些不是被否决，是**还没决定**，列在这里防止丢失（最终决策入 DECISIONS.md）：

### 议程 A · 跨客户数据是否聚合共用

是否允许把不同客户项目的数据聚合，做行业 pattern 分析？

- 支持: 强化数据飞轮、给新项目冷启动更多 prior
- 反对: 合规风险、客户合同条款限制
- 状态: 待咨询合同部门

### 议程 B · 是否做小红书爬虫扩充竞品数据

是否爬取竞品（其他公司投放的）笔记数据扩充数据库？

- 支持: 数据量级 10x，跨产品迁移更准
- 反对: 平台合规风险、数据质量参差不齐、tier 标签缺失（关键 label 没有）
- 状态: 暂不做（阶段 3 之前不考虑）

### 议程 C · Truth Vault 是否对外提供 API（商业化）

是否做成 SaaS 给其他 agency 用？

- 支持: 新业务线
- 反对: 客户数据隔离复杂、可能稀释帆谷自身优势
- 状态: 至少阶段 4 之后再考虑
